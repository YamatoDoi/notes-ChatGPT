# AGENTS.md

## プログラミングルール

可読性と保守性を重視し、コード全体として処理内容が読み手に伝わるようにしてください。
ただし、プログラマーが見て自明な処理や変数には、過剰なコメントを入れないでください。

### 1. ファイルヘッダ

各ファイルの冒頭には、そのファイルの目的や役割を簡潔に説明するコメントを記載してください。
ただし、Shebang、エンコーディング指定、ライセンス表記などが必要な場合は、それらの直後に記載してください。

### 2. Docstring

ユーザー定義関数にはDocstringを記載してください。
Docstringには必要に応じて、関数の目的、引数、戻り値、発生し得る例外、副作用を含めてください。
極めて短い内部補助関数など、説明が冗長になる場合は省略して構いません。

### 3. コメント

変数、定数、処理ブロックには、意図や役割が分かりにくい場合にコメントを追加してください。
単にコードを日本語に翻訳しただけのコメントは避け、「なぜその処理が必要なのか」を優先して説明してください。

### 4. 命名規則

変数名・関数名・クラス名・ファイル名は、役割が分かる具体的な名前を使用してください。
用途が明確な場合を除き、`tmp`、`data`、`value`、`result`、`test` のような曖昧な名前は避けてください。

### 5. 既存コードとの整合性

既存コードを修正する場合は、既存の命名規則、フォーマット、設計方針を尊重してください。
新旧のコードでスタイルが混在しないようにしてください。

### 6. 変更範囲

依頼内容に直接関係しない大規模なリファクタリングは行わないでください。
修正範囲は必要最小限に留め、既存機能への影響を避けてください。

### 7. 実装品質

マジックナンバーは可能な限り定数化してください。
重複コードは適切に関数化してください。
エラーハンドリングが必要な箇所では適切な例外処理を実装してください。
パフォーマンスよりも、まず可読性と保守性を優先してください。ただし、明らかに性能上の問題がある場合は改善案を提示してください。

### 8. コード生成時の説明

コードを生成または修正した際は、変更点、変更理由、既存動作への影響、注意点を簡潔に説明してください。
コード全体を再掲するのではなく、変更点を中心に説明してください。

## 技術ノート記事化ルール

ユーザーの依頼が「質問」で始まる場合、このリポジトリへ追加する技術ノート記事の作成依頼として扱います。

### 記事作成の基本方針

- 1ページ1テーマで、カテゴリ別ディレクトリに `index.html` を作成します。
- トップページの `index.html` に記事カードを追加します。
- 記事には、結論、背景、具体例、注意点、参照URLを含めます。
- 必要に応じてHTML/CSS/SVGで図解を作り、本文だけに依存しない説明にします。
- 仕様、料金、API、ライブラリ、法令、セキュリティなど変化し得る内容は、一次情報または信頼できる情報源を確認します。
- 調査に使ったURLは、記事末尾の「参照URL」に必ず残します。

### 公開前のガードレール

- 秘匿情報、個人情報、認証情報、未公開の社内情報を書かないでください。
- JSON、YAML、環境変数ファイル、鍵ファイルなど、秘密情報が入りやすいファイルはGit管理に含めません。
- 変更作業が完了したら、ユーザー確認を待たずにGitへコミットし、GitHubへpushします。
- コミット前に差分を確認し、AI生成内容が未確認の断定になっていないか確認します。
- ユーザーが「取り下げて」と依頼した場合は、履歴を書き換えずに `git revert` で直近の該当コミットを打ち消し、その取り下げコミットをpushします。

### HTML作成ルール

- ビルド不要で読める静的HTML、CSS、画像、SVGだけを使います。
- JavaScriptは必要性が明確な場合に限定します。
- スマホ幅でも本文、図、コードブロックが横にはみ出さないようにします。
- 見出し、リスト、表、図、コードブロックを適切に使い、あとから読み返しやすくします。
